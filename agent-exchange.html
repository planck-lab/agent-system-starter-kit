🤝 Agent Exchange: Multi-Agent via Git

A pattern for asynchronous collaboration between AI agents using Git as the communication layer. No framework, no message queue, no API — just markdown files in a shared repository.

The ProblemWhy Multi-Agent?

🧠

Cognitive Load

A research task takes 10 minutes. During that time, the agent can't answer chat messages.

🎯

Specialization

A coding agent needs different instructions than a research agent. Combining both dilutes quality.

🌙

Availability

An always-on server agent handles cron jobs at 3 AM. A laptop agent is available when you're working.

👁️

Review Bias

An agent reviewing its own output is like proofreading your own essay — it sees what it expects to see.

Sources: Talebirad & Nadiri (2023), Google A2A Protocol

The PatternHow It Works

agent-exchange/
├── signals/          # "Look at this"
│   ├── agent-a/
│   └── agent-b/
├── questions/       # "What do you think?"
├── responses/       # "Here's my take"
├── learnings/       # "I noticed something"
└── reviews/         # "I reviewed your work"

Rule: Each agent writes only in its own subfolder. Agents never modify each other's files.

Agent A writes signal → git push →→→ Agent B pulls
                                           Agent B reads, thinks, writes response
Agent A pulls ←←← git push ← Agent B

Timing: Fully asynchronous. Agent A pushes at 10:00, Agent B responds at 14:00, Agent A reads at 15:00. No need to be online simultaneously.

File Format

---
status: open           # open | in-progress | closed
by: agent-a            # who wrote this
date: 2026-03-25
---

# Title
Content in markdown.

In PracticeFour Use Cases

🔬 1. Research + Review

Agent A researches, Agent B challenges. Agent B has no investment in Agent A's output — it can be genuinely critical. Unlike self-review, which sees what it expects to see.

Flow: Human asks → Agent A researches → writes signal → Agent B reviews → finds 2 unsourced claims, 1 outdated price → Agent A fixes.

🌙 2. Day + Night Shifts

Agent A (laptop) is interactive during the day. Agent B (server) runs automated tasks overnight. At 08:00, results are waiting.

Flow: 22:00 write task list → 02:00 cron runs research → 08:00 pull results.

🔍 3. Build + Audit

One agent writes code or content. The other runs quality checks. Separation of concerns — the audit agent has different instructions focused on finding problems.

Flow: Agent A builds → signals "ready" → Agent B runs checks → writes review with findings.

📚 4. Learning Loop

Agents teach each other by documenting observations. Accepted learnings become design principles for both agents. Distributed knowledge creation.

Flow: Agent A notices pattern → writes learning → Agent B reviews: accepted/returned/rejected.

Honest AssessmentLimitations

LimitationDetail
Not real-timeGit push/pull has seconds-to-minutes latency
No streamingAgents see finished outputs only, can't watch each other think
Scale ceilingWorks for 2-5 agents. At 10+, consider message brokers
No routingAgents check for files via cron — no push notifications
Trust implicitAny agent with repo access reads everything
When NOT to use Exchange: Sub-second responses needed → A2A. 10+ agents → message broker. Untrusted agents → not designed for this. Binary data → Git handles markdown, not large files.

5 MinutesGetting Started

# Create exchange repo
mkdir agent-exchange && cd agent-exchange
git init
mkdir -p signals/agent-a signals/agent-b
mkdir -p questions/agent-a questions/agent-b
mkdir -p responses/agent-a responses/agent-b

# Create your first signal
cat > signals/agent-a/first-signal.md << 'EOF'
---
status: open
by: agent-a
date: 2026-03-25
---
# Hello from Agent A
Agent B, please acknowledge.
EOF

git add . && git commit -m "first signal"
# Push to shared remote (GitHub, private recommended)

The exchange/ folder in this kit is already structured. Connect two instances: clone on both machines, set up a cron for git pull, and each agent writes to its own subfolders.
